\section{Hoare Logic and Its Scalability Problem}

\subsection{Overview of Hoare Logic}
Hoare logic is a reasoning system for proving the specifications of programs.
It uses Hoare triples, written $\{P\} C \{Q\}$, to describe program specifications, where $C$ is the program, and $P$ and $Q$ are assertions that respectively describe pre- and post-conditions.
A Hoare triple is true if, whenever the precondition $P$ holds before executing the program $C$, the postcondition $Q$ will hold after executing $C$.
The Hoare logic provides rules for how to derive valid Hoare triples based on the structure of the program.

For example, the Hoare triple $\{ x = 0 \}\ x \defeq x + 1\ \{ x = 1 \}$
says that if $x$ is $0$ before executing the assignment $x := x + 1$,
then $x$ is $1$ afterward.

\subsection{Separation Matters in Program Verification}
For programs that manipulate heap-allocated data structures (like linked lists),
awareness of difference heap locations (pointers) is crucial for reasoning about
program behavior.
For example, if we want to reason about a list deallocation program,
\begin{figure}[H]
  
\begin{lstlisting}
LDispose(x) {
  while (x != null) {
    y := x;
    x := [x + 1];
    dispose(y);
    dispose(y + 1)
  };
  return null
}

\end{lstlisting}
\caption{List Dispose}
\end{figure}

the Hoare triple we can prove is:

$$
\vdash \{ \List(x) \} \LDispose(x) \{ x = \Null \}
$$

which means if $x$ points to a valid list before executing $\LDispose$, then
after executing $\LDispose$, $x$ will be null.

However, Hoare logic has scalability issues in a sense that it does not provide
a clear and simple way to extend already proven specifications with richer assertions.

For example, suppose we have proved the Hoare triple $\vdash \{ \List(x) \} \LDispose(x) \{ x = \Null \}$. 
How can we apply it to conclude the postcondition after executing $\LDispose$ in a program state that contains another $\List(y)$?

With a simple glance, the precondition is $\List(x) \land \List(y)$, and the
postcondition should be $x = \Null \land \List(y)$.
But this is not necessarily true, because $x$ and $y$ could alias!
We can conclude that $x = \Null$, but we cannot say anything about $y$ with certainty.

To make the specification describe the program correctly and precisely, we need
to use a more complex assertion that describes the separation of the two lists
in the precondition:

$$
(\List(x) \land \List(y)) \land
(\forall u.\ \reach(x, u) \land \reach(y, u) \Rightarrow u = \Null)
$$

The $\forall u.\ \reach(x, u) \land \reach(y, u) \Rightarrow u = \Null$ part of
the assertion says that the two lists $x$ and $y$ are separated, i.e., they do
not share any common nodes.

Only then can we conclude the postcondition $x = \Null \land \List(y)$.

If we have three lists $x$, $y$ and $z$, our precondition grows:

$$
\begin{array}{l}
(\List(x, \alpha) \land \List(y, \beta)) \\
\land\;(\forall u.\ \reach(x, u) \land \reach(y, u) \Rightarrow u = \Null) \\
\land\;\List(z) \\
\land\;(\forall u.\ \reach(z, u) \land (\reach(x, u) \lor \reach(y, u))
  \Rightarrow u = \Null)\,,
\end{array}
$$

as does the postcondition:

$$
x = \Null \land \List(y) \land \List(z) \land 
(\forall u.\ \reach(z, u) \land \reach(y, u) \Rightarrow u = \Null)\, .
$$

This approach does not scale; not only is it unclear where to add the separation assertion, but when we add more separation assertions, the complexity of the condition grows quadratically (as does the time needed to solve them).

\section{Separation Logic}
The scalability problem of Hoare logic is caused by the fact that the formulae in the Hoare triples always describe the \emph{entire} program heap.
Hence, when we want to extend the specification to describe more states, we must write more complex formulae that assert the separation of added states with the existing states.
Separation logic is a an approach that addresses this problem directly by introducing a new logical connective called the \textit{separating conjunction}, written $\star$.
Instead of describing the entire heap, a formula in separation logic describes only a \emph{part} of the heap, and uses $\star$ to combine two formulae that describe two strictly separate parts of the heap.
With this separating connective, we can extend the specification in a compositional way, without needing to write extra formulae that assert the separations.

\textbf{Note.} The separating conjunction operator bears superficial resemblance to the disjoint union; however, where the disjoint union re-indexes members of sets to reconcile intersection, the separating conjunction operator enforces that the two operands are disjoint.

\subsection{Syntax of Separation Logic Assertions}
\textbf{Assertions.}
The set of \emph{assertions}, $\Assert$, ranged over by
$P, P_1, Q, Q_1, \ldots$, is defined by:

\begin{align*}
P \in \Assert \ddefeq &
P \land P \mid P \Rightarrow P \mid \True \mid \False
& \text{classical connectives} \\
\mid\ &
E = E \mid E > E \mid E \in X \mid \cdots
& \text{boolean assertions} \\
\mid\ &
\exists x.\ P
& \text{quantification} \\
\mid\ &
P \star P \mid P \sepimp P \mid \Emp
\mid \displaystyle\mathop{\circledast}_{E_1 \le x < E_2} P
& \text{separating connectives} \\
\mid\ &
E \mapsto E
& \text{cell assertion}
\end{align*}

The syntax includes the classical logical connectives and quantifiers; the newly
added separating connectives are highlighted. 
Some connectives, such as $\forall$ and $\neg$, are not included in the syntax,
but they are derivable from the connectives given here.


\subsection{Meaning of Separation Logic Assertions}
What kind of program states does an assertion describe?
The meaning of the main assertions is given by the satisfaction relation $e, s,
h \models P$, which says that the assertion $P$ is satisfied in the program
state $(e, s, h)$, where $e$ is the environment of logical variables, $s$ is the
environment of program variables, and $h$ is the heap.

\begin{figure}      
\begin{align*}
e, s, h \models P_1 \land P_2
  &\Longleftrightarrow e, s, h \models P_1 \land e, s, h \models P_2 \\
e, s, h \models P_1 \Rightarrow P_2
  &\Longleftrightarrow e, s, h \models P_1 \Rightarrow e, s, h \models P_2 \\
e, s, h \models \True
  &\Longleftrightarrow \text{always} \\
e, s, h \models \False
  &\Longleftrightarrow \text{never} \\
e, s, h \models E_1 = E_2
  &\Longleftrightarrow (\Eval{E_1 = E_2} = \true) \land h = \emptyset \\
e, s, h \models E_1 > E_2
  &\Longleftrightarrow (\Eval{E_1 > E_2} = \true) \land h = \emptyset \\
e, s, h \models E \in X
  &\Longleftrightarrow (\Eval{E \in X} = \true) \land h = \emptyset \\
e, s, h \models \exists x.\ P
  &\Longleftrightarrow \exists v \in \Val.\ e[x \mapsto v], s, h \models P \\[1ex]
{\color{blue} e, s, h \models P_1 \star P_2}
  &{\color{blue}\ \Longleftrightarrow \begin{aligned}[t]
    & \exists h_1, h_2 \in \Heap.\ h = h_1 \uplus h_2 \land \\
    & e, s, h_1 \models P_1 \land e, s, h_2 \models P_2
    \end{aligned}} \\
{\color{blue} e, s, h \models \Emp}
  &{\color{blue}\ \Longleftrightarrow h = \emptyset} \\
{\color{blue} e, s, h \models \displaystyle\mathop{\circledast}_{E_1 \le x < E_2} P}
  &{\color{blue}\ \Longleftrightarrow \begin{aligned}[t]
    & \Eval{E_1} = i \in \mathbb{N}
      \land \Eval{E_2} = k \in \mathbb{N} \land \\
    & \bigl(i < k \Rightarrow
      \exists h_i, \ldots, h_{k-1}.\ h = h_i \uplus \cdots \uplus h_{k-1}
      \land \\
    & \qquad \forall j.\ i \le j < k.\ e, s, h_j \models P[j/x]\bigr) \land \\
    & \bigl(i \ge k \Rightarrow h = \emptyset\bigr)
    \end{aligned}} \\
{\color{blue} e, s, h \models E_1 \mapsto E_2}
  &{\color{blue}\ \Longleftrightarrow h = \{\Eval{E_1} \mapsto \Eval{E_2}\}} \\
{\color{blue} e, s, h \models \Pred(\vec{E})}
  &{\color{blue}\ \Longleftrightarrow \cdots}
\end{align*}
\caption{Satisfaction Relation for Separation Logic Assertions}
\label{satisfaction}
\end{figure}

\textbf{Note.} One can think of $P_1 \star P_2$ as separation, and $P_1 \land P_2$ as overlap.
So, for example, $10 \mapsto 1 \star 10 \mapsto 2$ is unsatisfiable, as separate heaps cannot have any intersection in their domains---conversely, $10 \mapsto 1 \land 11 \mapsto 1$ is unsatisfiable because the two heaps do not have exactly identical domains.
Note that $\land$ might superficially seem equivalent to $=$, but the distinction is best revealed by the idiom $P \star \True$.

For example, $(10 \mapsto 1 \star \True) \land (11 \mapsto 2 \star \True)$ is
satisfiable by heap $\{10 \mapsto 1, 11 \mapsto 2\}$, because $\True$ represents
any heap (analagously to how $10 \mapsto 1 \land 10 \mapsto \_$ is satisfiable
by heap $\{10 \mapsto 1\}$, because $\_$ is any value).
Roughly, instead of ``are these things equal,'' it's ``can these things be made
equal.''

\subsection{Separation Logic Rules}

\begin{figure}
\begin{mathpar}
\inferrule*[right=\textsc{assign}]
  {x \notin \pv(E')}
  {\triple{x = E' \star E \in \Val}{x := E}{x = E[E'/x]}}
\and
\inferrule*[right=\textsc{lookup}]
  {x \notin \pv(E')}
  {\triple{x = E' \star E \mapsto E_1}
          {x := [E]}
          {x = E_1[E'/x] \star E[E'/x] \mapsto E_1[E'/x]}}
\and
\inferrule*[right=\textsc{mutate}]
  { }
  {\triple{E_1 \mapsto E \star E_2 \in \Val}{[E_1] := E_2}{E_1 \mapsto E_2}}
\and
\inferrule*[right=\textsc{new}]
  {x \notin \pv(E')}
  {\triple{x = E' \star E \in \mathbb{N}}
          {x := \mathsf{new}(E)}
          {\exists y.\ x = y \star
           \mathop{\circledast}_{0 \le i < E[E'/x]}((y + i) \mapsto \Null)}}
\and
\inferrule*[right=\textsc{dispose}]
  { }
  {\triple{E \mapsto E_1}{\mathsf{dispose}(E)}{\Emp}}
\and
\inferrule*[right=\textsc{skip}]
  { }
  {\triple{\Emp}{\mathsf{skip}}{\Emp}}
\and
\inferrule*[right=\textsc{seq}]
  {\triple{P}{C_1}{Q} \\ \triple{Q}{C_2}{R}}
  {\triple{P}{C_1;\ C_2}{R}}
\and
\inferrule*[right=\textsc{if}]
  {\triple{P \land E}{C_1}{Q_1} \\
   \triple{P \land \neg E}{C_2}{Q_2}}
  {\triple{P \star E \in \Bool}
          {\mathsf{if}\ (E)\ C_1\ \mathsf{else}\ C_2}
          {Q_1 \lor Q_2}}
\and
\inferrule*[right=\textsc{while}]
  {\triple{P \land E}{C}{P \star E \in \Bool}}
  {\triple{P \star E \in \Bool}
          {\mathsf{while}\ (E)\ C}
          {P \land \neg E}}
\and
\inferrule*[right=\textsc{frame}]
  {\triple{P}{C}{Q} \\
   \Mod(C) \cap \pv(R) = \emptyset}
  {\triple{P \star R}{C}{Q \star R}}
\and
\inferrule*[right=\textsc{cons}]
  {\models \blueassn{P \Rightarrow P'} \\
   \triple{P'}{C}{Q'} \\
   \models \blueassn{Q' \Rightarrow Q}}
  {\triple{P}{C}{Q}}
\and
\inferrule*[right=\textsc{exists}]
  {\triple{P}{C}{Q}}
  {\triple{\exists x.\ P}{C}{\exists x.\ Q}}
\and
\inferrule*[right=\textsc{disj}]
  {\triple{P_1}{C}{Q} \\ \triple{P_2}{C}{Q}}
  {\triple{P_1 \lor P_2}{C}{Q}}
\end{mathpar}
\caption{Separation Logic Rules}
\label{rules}
\end{figure}

So far \cref{satisfaction} defines the programs that satisfy a separation logic
Hoare triple.
Now we define how to derive such triples from program syntax in \cref{rules}. 

The soundness condition defined in \cref{soundness} guarantees that the derivation rules are
correct (the proof is omitted here). 

\begin{figure}
      
\[
\begin{aligned}[t]
\forall P, C, Q, e, s, h.&
\vdash \{P\}\ C\ \{Q\}
\Longrightarrow e, s, h \models P \\
& \Longrightarrow
\left(
  \neg\bigl((s,h), C \Downarrow \lightning\bigr)
  \land
  \left(
    \forall s', h'.\ 
    (s,h), C \Downarrow (s',h')
    \Longrightarrow e, s', h' \models Q
  \right)
\right)
\end{aligned}
\]
\caption{Soundness of Separation Logic}
\label{soundness}
\end{figure}

It says if we can infer Hoare triple $\vdash \{P\}\ C\ \{Q\}$ from the program
$C$, then whenever we run the $C$ with a state $s, h$ that satisfies the
precodition $P$, after executing the command $C$ the outcome states $s', h'$
satisfies the postcondition $Q$.
