\section{Compositional Symbolic Execution on WISL}

The linear memory model of the WHILE language, given in the previous lecture has been useful demonstrator memory model for introducing separation logic. However, it has some big caveats such as generating way too numerous constraints on buffers regarding pointer arithmetic and buffer separation. \\

One way in which analysis tools for real world languages reason about memory odels in a tractable way, is by viewing the heap not as a map index by naturals, but rather as a collection of memory blocks, and treating pointers as block-offset pairs. Thus, we introduce WISL, a more general language that's still very close to WHILE, and which satisfies this memory model.  \\


\[
\hat{\sigma} = (\hat{s}, \hat{\mu}, \widehat{\Pred}, \hat{\pi})
\]
where $\hat{s} : \mathsf{PVar} \rightharpoonup_{\mathrm{fin}} \mathsf{LExp}$ is the symbolic store, $\hat{\mu} : \mathsf{LExp} \rightharpoonup_{\mathrm{fin}} \mathsf{LExp}$ is the symbolic linear heap, $\widehat{\Pred}$ stores user-defined predicate instances, and $\hat{\pi}$ is the symbolic path condition.\\

For Gillian to reason about (memory) models such as linear models (with/out unique branch matching/cut branching/negative informartion $\dots$),fractional ownership models, OOP models, and many (many) more, we need to establish soundness conditions that can relate concrete with symbolic models.


\subsubsection*{The Over-Approximation (OX) Soudness Codition}

OX Soundness enforces that all states reachable by concrete execution, are also reachable by symbolic execution (which allows for verifying software):

\[
\sigma, C \reduces \sigma' \wedge \sigma \models \hat{\sigma} \implies \exists \hat{\sigma}', \hat{\sigma},C \reduces \hat{\sigma}' \wedge \sigma' \models \hat{\sigma}' 
\]

\subsubsection*{The Under-Approximation (UX) Soudness Codition}

UX Soundness enforces the opposite condition, and thus it's particulartly addequate for bug finding (i.e: ensure that a bug will arise in the concrete model provided that it shows in the symbolic one):

\[
\hat{\sigma},C \reduces \hat{\sigma}' \wedge \sigma' \models \hat{\sigma}' \implies \exists \sigma',  \sigma, C \reduces \sigma' \wedge \sigma \models \hat{\sigma} 
\]

\section{Extending Proof Rules to WISL}

To formally reason beyond proof sketches, we also need to extend the proof rules we saw in the previous lectures. For that, we'll use the notation $\mu.\alpha(\vec{v})\rightsquigarrow_{\mathcal M} o':(\mu',\vec v')$ for an action $\alpha$, which is run over the memory $\mu$ and uses the parameters $\vec{v}$ and results in an outcome $o$ (which can be $ok$, $error$, $miss$...), a memory $\mu'$ and return arguments $\vec v'$.

\begin{figure}[H]
\begin{tikzpicture}[font=\Large]
 
% ================= OX frame =================
\begin{scope}
  \ruleframe{17.0}{3.75}{0.9}
    {OX frame}{2.7}
    {``If memory is removed by an action, (non-missing)\ \ behaviour is unaffected''}
    {$\begin{array}[t]{@{}ll@{}}
       \text{If} & (\mu\cdot\mu_f).\alpha(\vec v)\rightsquigarrow_{\mathcal M} o:(\mu',\vec v')\\[8pt]
       \text{then} & \exists\,\mu'',\vec v'',o'.\ \mu.\alpha(\vec v)\rightsquigarrow_{\mathcal M} o':(\mu'',\vec v'')\ \text{and}\\[8pt]
       & \quad(o'\neq miss \Rightarrow (o'=o\ \text{and}\ \vec v''=\vec v'\ \text{and}\ \mu'=\mu''\cdot\mu_f))
     \end{array}$}
\end{scope}
 
% ================= UX frame =================
\begin{scope}[shift={(0,-4.1)}]
  \ruleframe{17.0}{3.3}{0.9}
    {UX frame}{2.7}
    {``If memory is added by an action, (non-missing) behaviour is unaffected''}
    {$\begin{array}[t]{@{}ll@{}}
       \text{If} & \mu.\alpha(\vec v)\rightsquigarrow_{\mathcal M} o:(\mu',\vec v')\ \text{and}\ o\neq miss\ \text{and}\ \mu'\cdot\mu_f\ \text{is defined}\\[8pt]
       \text{then} & (\mu\cdot\mu_f).\alpha(\vec v)\rightsquigarrow_{\mathcal M} o:(\mu'\cdot\mu_f,\vec v')
     \end{array}$}
\end{scope}
 
\end{tikzpicture}
\end{figure}

Although the UX frame rule is fairly straight-forward (we can always add more memory $\mu_f$ without affecting behaviour as long as the action didn't resulted in a $miss$, which is correct, since if we could procure a bug using $\mu$ memory, we cal also procure it using more than $\mu$), the OX frame rule is more subtle, since to capture that we can extend
analysis results to larger states we must say, perhaps counterintuitively, that removing a "frame" $\mu_f$ from $\mu \cdot \mu_f$ results in either a $miss$ outcome or the same behaviour as executing from the full
state, rather than more straightforwardly stating something about adding more state\\

Moreover, we also need to "lift" the soundness condition into our new context (which is more easily expressed as making the following diagrams commute):
